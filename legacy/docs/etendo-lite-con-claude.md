# Etendo Lite — What we built and how we did it with Claude

## Context

Etendo has a Headless API (EtendoRX) that exposes ERP data via REST endpoints with JWT authentication. The API is powerful but verbose: to create a sales invoice you need to chain at least four calls, manage internal IDs, resolve taxes, and know the exact names of endpoints and fields.

The idea was to build a middleware + frontend layer that abstracts all that complexity and exposes only what a business user needs, with an interface similar to Holded.

---

## What we built

### 1. Express Middleware (`/server`)

A lightweight REST API that acts as an orchestrator between the frontend and EtendoRX Headless.

**Implemented endpoints:**

| Method | Route | Description |
|--------|------|-------------|
| POST | `/lite/auth/login` | Login and get JWT |
| GET | `/lite/masters/customers` | List customers |
| POST | `/lite/masters/customers` | Create customer + automatic address |
| GET | `/lite/masters/vendors` | List vendors |
| POST | `/lite/masters/vendors` | Create vendor + automatic address |
| GET | `/lite/masters/products` | List products |
| POST | `/lite/masters/products` | Create product |
| GET | `/lite/taxes` | List tax categories |
| GET | `/lite/sales/orders` | List sales orders |
| POST | `/lite/sales/orders` | Create and confirm sales order |
| GET | `/lite/sales/invoices` | List sales invoices |
| POST | `/lite/sales/invoices` | Create sales invoice with lines |
| GET | `/lite/payables/payables` | Pending purchase invoices |
| GET | `/lite/payables/receivables` | Pending sales invoices |
| GET | `/lite/payables/summary` | Consolidated payable/receivable summary |
| GET | `/docs` | Autogenerated Swagger UI |

**What the middleware resolves for you:**
- JWT authentication propagated in every request
- Multi-tenant (configuration via JSON file in `/tenants/`)
- Complex orchestration: for example, creating an invoice implies resolving the customer's address, creating the header, and creating the lines in parallel
- Automatic tax resolution from the product category
- OpenAPI documentation automatically generated from the code (swagger-autogen)

### 2. React Frontend (`/ui`)

SPA with full routing, no external UI dependencies. All inline styles.

**Views:**
- **Dashboard** — real-time summary of payables/receivables
- **Customers / Vendors** — table + inline creation
- **Products** — table + inline creation
- **Sales Orders** — form with customer and product search
- **Sales Invoices** — list with status badges
- **Purchase Invoices** — list with expiration alerts

## The most interesting technical problem

### Hibernate lazy-loading bug in EtendoRX

When trying to create customers via the `BPCustomer` endpoint, the server returned:

```
could not initialize proxy [ADReference#166] - no Session
```

**Root of the problem:** `DataSourceServlet` in EtendoRX serializes the POST response *outside* the active Hibernate session. If the endpoint exposes fields that reference `ADReference` entities (system enums), Hibernate tries to load those lazy proxies and fails because there is no longer an open session.

**Fix:** The `etrx_openapi_field` table controls which fields each endpoint exposes. If it has no rows for a tab, it exposes everything — which increases the risk of touching problematic references.

Solution: add explicit rows in `etrx_openapi_field` with only the necessary fields. This restricts what Hibernate loads during serialization.

```sql
-- Example: restrict BusinessPartner to the minimum fields for creation
INSERT INTO etrx_openapi_field (etrx_openapi_field_id, ad_tab_id, ad_field_id, ...)
VALUES (...);  -- Organization, Name, SearchKey, BPCategory, TaxID, Language
```

Requires Tomcat restart to apply.

This kind of diagnosis usually takes hours: reading Tomcat logs, understanding the EtendoRX stack, searching the Java code of `DataSourceServlet`. With Claude, we solved it in minutes.

## Claude's contribution to development

### Speed of investigation

Claude read and reasoned about the EtendoRX stack (Java/Hibernate) to identify the root cause of the bug without needing a debugger or having the full source code. It made hypotheses, validated them against the logs, and proposed the correct fix on the first try.

### Orchestration of complex code

The `POST /lite/sales/invoices` endpoint requires:
1. Searching for the customer's billing address (`BPAddress` with `invoiceToAddress==true`)
2. Creating the invoice header (`SalesInvoice`)
3. For each line: resolving the tax ID from the product category, and creating the line (`SalesInvoiceLine`) — all in parallel

Claude wrote all that orchestration logic correctly the first time, including error handling and state propagation.

### Accumulated context

Throughout the session, Claude maintained context of all the API's peculiarities:
- The `BPCustomer` vs `BusinessPartner` endpoint (tab 223 vs 220)
- That `PurchaseOrderLines` uses `salesOrder` as FK (naming quirk)
- That POST responses have `status` at the root, not in `response`
- That `SalesInvoice` had a broken ADField in the installation

Without that accumulated memory, every developer touching the API starts from scratch.

### Complete frontend in one iteration

The frontend — sidebar, routing, 7 pages, shared design system, forms with typeahead search, tables with loading states — was generated in a single prompt with the correct result compiling without TypeScript errors.

## ChatGPT (web) vs Claude CLI — the difference in practice

Using ChatGPT in the browser is the analogy of having a very capable consultant next to you: you can ask them anything, they give good advice, they write code snippets for you. But the consultant doesn't touch your computer. You are the one who copies, pastes, saves files, runs commands, reads errors, and copies again. The loop is: ask → copy → paste → error → copy the error → ask → copy → paste.

Claude CLI operates differently. It has direct access to the filesystem, it can read files, edit them, run commands, read the output, and chain all that without intervention. The loop is: instruction → done.

### Concrete differences in this project

| | ChatGPT web | Claude CLI |
|---|---|---|
| Reads existing code before modifying | No (only what you paste) | Yes (direct access) |
| Maintains context across files | No | Yes (read etendo.js, sales.js, and frontend in the same session) |
| Executes and sees the result | No | Yes (ran TypeScript builds, saw errors in real time) |
| Corrects its own errors | Manual (you pass the error) | Automatic (reads output and adjusts) |
| Speed of development loop | Slow (human in the middle) | Fast (the cycle is the instruction) |

### What changed in practice

The Hibernate lazy-loading bug is a good example. With ChatGPT:
- You copy the stack trace
- It explains what it could be
- You propose a hypothesis
- You search the code
- You copy the relevant code
- Adjust the solution
- You copy the resulting SQL
- You execute it yourself

With Claude CLI:
- You describe the error to it
- It reads the server logs (direct access)
- It reads `DataSourceServlet.java` and `etrx_openapi_field`
- Proposes the complete hypothesis with context
- Writes the SQL fix directly
- All in the same flow

The difference is not that one is smarter than the other. The difference is that with Claude CLI **the developer stops being the connector between the AI and the codebase**. That changes how much can be done in one session.

### What role does the developer play

With ChatGPT the developer is still the main executor — the AI assists.
With Claude CLI the developer becomes the one who defines goals and reviews results — the AI executes.

For exploration work, research or specific questions: ChatGPT web is enough.
For building real software on an existing codebase: the difference is hours vs days.
